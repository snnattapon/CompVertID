## ระดับความเชื่อมั่น

CompVertID ให้คะแนนความเชื่อมั่นแต่ละผลลัพธ์ 12 ระดับ จัดกลุ่มด้วยสี

| Confidence | สี | ความหมาย |
| --- | --- | --- |
| `verified_high` | 🟢 | มี KEGG ID อยู่แล้ว และสูตรตรงกับ KEGG |
| `found_high` | 🟢 | ค้นเจอจากชื่อ และสูตรตรง |
| `verified_no_formula` | 🟡 | มี KEGG ID แต่ไม่มีสูตรมาเทียบ |
| `found_no_formula_check` | 🟡 | ค้นเจอ แต่ไม่มีสูตรมาเทียบ |
| `name_match_verify` | 🟡 | ชื่อตรงเป๊ะกับ KEGG แต่ยังไม่ได้ยืนยันด้วยสูตร |
| `charge_diff` | 🟡 | สูตรต่างแค่ไฮโดรเจน/ประจุ (เช่น รูป protonated) — ผลยังเชื่อถือได้ |
| `ambiguous` | 🟠 | ค้นเจอหลายตัวที่สูตรตรงหมด ต้องเลือกเอง |
| `formula_mismatch` | 🟠 | สูตรต่างที่โครงคาร์บอน — อาจเป็นคนละสาร |
| `low` | 🟠 | ค้นเจอแต่ไม่มีตัวไหนสูตรตรง |
| `not_found` | ⚪ | ค้นไม่พบใน KEGG |
| `not_in_KEGG` | ⚪ | เป็น lipid species ที่ KEGG ไม่ครอบคลุม |
| `no_name` | ⚪ | ไม่มีชื่อสารในข้อมูลเข้า |

### จุดที่ควรเข้าใจ: `charge_diff` กับ `formula_mismatch` ต่างกัน

สองอันนี้แยกกันโดยตั้งใจ เพราะความหมายต่างกันมาก:

- **`charge_diff`** — สูตรต่างแค่ไฮโดรเจน (H±1, H±2) มักเกิดจากสภาพประจุ (เช่น สาร quaternary ammonium ที่ KEGG เก็บในรูป protonated) **ผลยังเชื่อถือได้** จัดเป็นสีเหลือง
- **`formula_mismatch`** — สูตรต่างที่ธาตุอื่นหรือโครงคาร์บอน เช่น มีคาร์บอนต่างกัน หรือมี Na เพิ่ม **อาจเป็นคนละสาร** ควรตรวจสอบ จัดเป็นสีส้ม

## การทำงานเบื้องหลัง

เมื่อประมวลผลสารแต่ละตัว CompVertID จะ:

1. **ถ้ามี KEGG ID อยู่แล้ว** → ดึงข้อมูลจาก KEGG มายืนยัน เทียบสูตร
2. **ถ้าไม่มี KEGG ID** → ค้นจากชื่อสาร
   - ถ้าเป็น lipid species (เช่น `PC(18:1/0:0)`) → ข้าม เพราะ KEGG ไม่ครอบคลุม
   - ถ้าเจอตัวเดียว → เทียบสูตรยืนยัน
   - ถ้าเจอหลายตัว → กรองด้วยสูตร ถ้าเหลือตัวเดียวคือคำตอบ ถ้าเหลือหลายตัวคือ `ambiguous`
3. **รวมผล pos + neg** → สารที่ได้ KEGG ID เดียวกันจะถูกยุบเป็นแถวเดียว (Mode = `pos;neg`)

การเรียก KEGG API มีการ **cache** (ไม่เรียกซ้ำในการรันเดียวกัน) และ **retry** อัตโนมัติเมื่อเครือข่ายมีปัญหา

## เรื่อง log และความเป็นส่วนตัว

ทุกครั้งที่รัน CompVertID จะสร้างไฟล์ log หนึ่งไฟล์ (ชื่อ `compvertid_run_<วันเวลา>.log`) เก็บไว้ **ในโฟลเดอร์เดียวกับไฟล์ผลลัพธ์**

**โหมดปกติ** log เก็บเฉพาะ:
- เวลาเริ่ม/จบ, จำนวนแถว, เวอร์ชันโปรแกรม, ระบบปฏิบัติการ
- สถิติการเรียก KEGG API (เป็นตัวเลข)
- ข้อความ error (ถ้ามี)

log โหมดปกติ **ไม่เก็บชื่อหรือสูตรของสารประกอบ** เพื่อความเป็นส่วนตัวของงานวิจัย

**โหมด debug** (เปิดด้วย `--debug` หรือ checkbox ใน GUI) จะเพิ่มการบันทึกรายละเอียดของ **เฉพาะสารที่มีปัญหา** เพื่อช่วย debug — เปิดเมื่อคุณตั้งใจจะส่ง log ให้ผู้พัฒนาตรวจสอบเท่านั้น

### เมื่อเจอปัญหาและอยากขอความช่วยเหลือ

1. เปิดโหมด debug แล้วรันซ้ำ
2. เปิดไฟล์ log อ่านดูก่อน (เพื่อความสบายใจว่าจะแชร์อะไร)
3. ส่งไฟล์ log ให้ผู้พัฒนา

## หนึ่งการทดลองต่อการรัน

### ทำไมต้องระวัง

ช่อง Positive/Negative mode ออกแบบมาสำหรับ pos/neg ของ **ตัวอย่างชุดเดียวกัน** เมื่อรวมผล โปรแกรมจะยุบสารที่มี KEGG ID เดียวกันเป็นแถวเดียว โดยถือว่าเป็นสารเดียวกันที่เจอทั้งสอง mode

ถ้านำไฟล์คนละโปรเจกต์มาใส่ สารที่บังเอิญซ้ำกันจะถูกยุบรวม ทำให้ **ข้อมูลปนกันโดยไม่มีการแจ้งเตือน** (ไม่มี error ให้เห็น แต่ผลลัพธ์ผิดความหมาย)

### วิธีที่ถูกต้อง

- หนึ่งโปรเจกต์/การทดลอง = หนึ่งการรัน
- pos และ neg ต้องมาจากตัวอย่างชุดเดียวกัน
- มีหลายโปรเจกต์ → รันแยกทีละอัน เก็บผลแยกไฟล์

## สำหรับผู้ร่วมพัฒนา

### โครงสร้างโค้ด

```
src/compvertid/
├── config.py          ค่าคงที่ทั้งหมด (API, ชื่อคอลัมน์, ตารางสี, ลำดับความเชื่อมั่น)
├── kegg_api.py        คุยกับ KEGG REST API + cache + retry (ไม่มี UI)
├── core.py            ตรรกะ mapping ล้วน ๆ (ไม่มี UI/print/path)
├── input_loader.py    อ่านไฟล์เข้า (xlsx / txt / csv) แปลงเป็นรูปแบบเดียว
├── logging_setup.py   ระบบ log (rotation, privacy, debug)
├── cli.py             ทางเข้าแบบ command line
├── gui.py             ทางเข้าแบบหน้าต่าง (tkinter)
└── __init__.py        public API
```

### หลักการออกแบบ

- **แยกตรรกะออกจากการแสดงผล** — `core.py` และ `kegg_api.py` ไม่มี `print`/UI เลย รายงานผลผ่าน callback ที่ CLI/GUI ส่งเข้ามา ทำให้ทดสอบง่ายและใช้ซ้ำได้ทั้งสอง UI
- **ค่าคงที่รวมที่เดียว** — แก้ `config.py` ที่เดียว มีผลทั้งระบบ
- **ตรวจสอบ input ก่อน** — เช็คคอลัมน์ที่จำเป็นก่อน map แจ้ง error ที่อ่านเข้าใจได้

### การรันเทสต์

```bash
python smoke_test.py              # ทดสอบ core + pipeline (ไม่ต้องต่อเน็ต)
python false_alarm_analysis.py    # ดูว่าระบบ flag เคสต่าง ๆ อย่างไร
python gui_headless_test.py       # ทดสอบ GUI (ต้องมี display หรือ Xvfb)
```

## การ build

### ข้อจำกัดสำคัญ

PyInstaller **ไม่สามารถ build ข้าม OS ได้** — ต้อง build บน OS เป้าหมายจริง:
- `.exe` (Windows) ต้อง build บน Windows
- `.app`/binary (macOS) ต้อง build บน macOS
- แต่ละสถาปัตยกรรมชิป (Intel / Apple Silicon) ต้อง build บนชิปนั้น

เราจึงใช้ **GitHub Actions** ที่มี runner ครบทุก OS build ให้อัตโนมัติ

### build อัตโนมัติผ่าน GitHub Actions

Workflow `.github/workflows/build.yml` จะทำงานเมื่อ push tag เวอร์ชัน:

```bash
git tag v0.1.0
git push origin v0.1.0
```

มันจะ build ทั้ง 3 เป้าหมาย (Windows, macOS Intel, macOS Apple Silicon) แล้วแนบไฟล์เข้ากับ GitHub Release อัตโนมัติ

### build เองในเครื่อง (ทดสอบ)

```bash
pip install -e ".[build]"
pyinstaller build/compvertid.spec --distpath build/dist
```

จะได้ไฟล์ในโฟลเดอร์ `build/dist/` (เป็น format ของ OS ที่รัน)

## การดูแลรักษา

### Dependabot

repo นี้เปิด [Dependabot](.github/dependabot.yml) ไว้ มันจะเปิด Pull Request มาให้เมื่อมีเวอร์ชันใหม่ของ:
- GitHub Actions ที่ใช้ (เช่น `actions/checkout`)
- Python dependencies (pandas, requests ฯลฯ)

**หมายเหตุ (2026):** คำสั่งแบบพิมพ์ comment ใน PR เช่น `@dependabot merge` ถูกยกเลิกแล้ว — ให้ใช้ปุ่มใน GitHub UI แทน (กด Merge ตามปกติ)

### เรื่องเวอร์ชันของ GitHub Actions

Workflow ปักเวอร์ชันแบบ **major เท่านั้น** (เช่น `actions/checkout@v5`) ซึ่งจะได้ตัวแก้บั๊กย่อย (patch/minor) อัตโนมัติ แต่ไม่โดน major ใหม่ที่อาจ break

ควรคอยดู **deprecation notice** ในหน้า Actions ของ GitHub — เมื่อ action ที่ใช้กำลังจะตกรุ่น (เช่นเรื่อง Node.js runtime) ให้อัป major version ตาม หรือรอ Dependabot เปิด PR มาให้
